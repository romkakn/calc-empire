Editing & Operating Guide

A no-code reference for running Calc Empire. Written for an SEO expert who doesn't code. Every workflow ends with either a GitHub edit (Vercel auto-deploys in ~30 sec) or a Vercel env var change (then Redeploy).

ℹ
Two URLs you live in:
  • GitHub repo (source of truth): github.com/romkakn/calc-empire
  • Vercel dashboard (deploys + env vars + analytics): vercel.com/roma-s-projects3/calc-empire

1. The mental model

Three layers, from where you spend your time to where the machinery hums silently:

  1. You edit a file on GitHub (or you tell Claude what to change). Files are mostly plain text with familiar Markdown-ish content inside.
  2. GitHub stores the change and emits a webhook to Vercel. You see your edit as a new commit on the main branch.
  3. Vercel rebuilds the site from the latest commit and swaps the live URL to the new build. ~30 seconds end-to-end.

You don't touch a server, install anything, or run any commands. Everything is either a GitHub UI click or a Vercel UI click.

2. Edit page content

Every calculator page lives in two files under src/app/<slug>/:

page.tsx
The visible page — title, tagline, FAQs, references, worked example, common mistakes. Edit text here.
Calculator.tsx
The interactive widget — input labels, helper text, the formula in code. Touch only if you understand the math.

Edit a FAQ on GitHub

  1. Open src/app/mortgage-recast-calculator/page.tsx (any calc's page.tsx).
  2. Click the pencil icon (✎) at the top-right of the file view.
  3. Find the FAQS array near the top. Each item looks like:
{
  question: "What are the downsides of a mortgage recast?",
  answer:
    "You give up liquid cash for a lower payment, the rate…",
},
  1. Edit any question or answer string between the quotes. Don't change the commas, braces, or quotes around them.
  2. Scroll to the bottom → Commit changes → optionally add a message → Commit changes again.
  3. Wait 30 sec. Refresh the live page. Done.
⚠
The one rule that'll trip you up: keep the surrounding punctuation intact. If you delete a comma at the end of an item or a quote around a string, the build fails. Vercel emails you the build error so you know to fix it.

Edit a worked example, references, or related terms

Same file, same pattern. Look for the array named after what you want to change (e.g. scenarios, references, terms). Each item is a small object with English strings inside quotes. Edit the strings, commit, done.

Edit the homepage tagline

Open data/calculators.json → change "tagline" under site. Commit. The homepage h1, <title>, meta description, OG/Twitter tags, and JSON-LD all update automatically.

3. SEO controls

Everything an SEO touches lives in a small, predictable set of places.

Per-page meta tags

Each page's page.tsx exports a metadata object near the top:

export const metadata: Metadata = {
  title: "Mortgage Recast Calculator (2026)",
  description: "Estimate your new monthly payment…",
  alternates: { canonical: "/mortgage-recast-calculator" },
  openGraph: { title, description, url, type: "article" },
  twitter:   { card: "summary_large_image", title, description },
};
title
≤ 60 chars. Browser tab + SERP + OG title.
description
≤ 155 chars. Meta description + OG/Twitter description.
alternates.canonical
Always exactly matches the route. Don't change unless you move the page (then also add a redirect).

Sitemap & robots

Both are auto-generated from data/calculators.json. To add a page to /sitemap.xml, mark its entry "status": "live". To remove it, set to "in-progress" or "planned".

robots.txt is a static Allow: / with the sitemap link. To noindex a single page, add robots: { index: false } to its metadata block (we do this for /paycheck-calculator preview).

JSON-LD structured data

Every calc page emits 5 schemas. They're generated from helpers in src/lib/schema.ts — you don't hand-write them:

SoftwareApplication
Required Calculator schema. Reads name + category + description from the page.
HowTo
Steps for using the calculator. Read directly from the page's metadata.
FAQPage
Generated from the same FAQS array you edit for the visible FAQ.
BreadcrumbList
Home → category → page.
Person
Author (you). Edit data/calculators.json → site.author.

To validate any page's structured data: validator.schema.org → paste a live URL.

Open Graph image & favicon

Auto-generated:

opengraph-image.tsx
1200×630 PNG built at request time. Used for every page unless that page ships its own. Edit the JSX in src/app/opengraph-image.tsx to change the layout.
icon.tsx
32×32 favicon. Edit src/app/icon.tsx to change the colour or letters.

Install header codes (GA / GTM / FB Pixel / GSC)

You don't edit code for these. Set environment variables in Vercel → Redeploy. The layout reads the env vars and injects the right scripts only if set.

Submit the sitemap

  1. Verify ownership in GSC (above).
  2. GSC → Sitemaps → enter https://<your-domain>/sitemap.xml → Submit.
  3. Google starts crawling within ~24 hours. Watch Coverage for indexed pages.
  4. Repeat at Bing Webmaster Tools (optional but free).

4. Vercel essentials

Vercel hosts the live site, runs every build, and serves the analytics dashboard. You almost never log in unless something changed in the plumbing.

The five Vercel tabs

Overview
Latest deployment + live URL. The default tab.
Deployments
History of every build. Green = Ready. Red = Error. Blue = Building. Click any row to see logs. Click ⋯ → Promote to Production to instantly roll back.
Analytics
Visitor numbers (page views, top pages, top referrers). Free on Hobby plan up to 2,500 events/month — enable in this tab.
Speed Insights
Core Web Vitals from real users. Auto-enabled once the site has traffic.
Settings
Env vars · Domains · Git integration. Where 95% of your Vercel time goes.

Environment variables

The home for header codes, verification tags, and the production URL.

  1. Settings → Environment Variables.
  2. Click Add.
  3. Key = the name (e.g. NEXT_PUBLIC_GA_MEASUREMENT_ID). Value = the actual code.
  4. Environments → tick Production.
  5. Save.
  6. Deployments tab → ⋯ on the latest → Redeploy. The new value goes live in ~30 sec.

How auto-deploy works

Every push to main triggers a build. Two paths fire on every push for redundancy:

  1. Vercel's GitHub App webhook — the native path. Fast.
  2. A backup GitHub Action at .github/workflows/vercel-deploy.yml — pings a Vercel deploy hook URL stored as a repo secret. Fires even if Vercel's webhook hiccups.
⚠
The commit-author-email rule: Vercel blocks any deployment whose commit author email it can't recognise. Your repo's user.email is set to seomanager211@gmail.com to match your Vercel account. If you ever commit from a different machine, run git config user.email seomanager211@gmail.com first, or you'll see BLOCKED deployments.

Custom domain

  1. Buy a domain (Namecheap, Porkbun, Google Domains). ~$10/yr.
  2. Vercel → Settings → Domains → Add → type the domain.
  3. Vercel shows DNS records to add at your registrar. Copy + paste them.
  4. Wait 5–60 min. Certificate auto-issues.
  5. Edit env var NEXT_PUBLIC_SITE_URL to the new domain. Redeploy.

The sitemap, robots.txt, JSON-LD canonical URLs, and OG tags all rewrite to the new domain automatically once the env var is updated + redeployed.

Rollback

Deployments tab → find any green "Ready" row from before things broke → click ⋯ → Promote to Production. Live within 2 seconds. No build required.

Preview deployments

Every branch other than main gets its own preview URL when pushed. Format: calc-empire-git-<branch>-roma-s-projects3.vercel.app. Use to test changes before merging into main.

5. GitHub essentials

Edit a file from the web UI

  1. Open the file on github.com (use a direct link from this guide).
  2. Click the pencil icon ✎ top-right.
  3. Edit. Preview tab shows rendered Markdown if applicable.
  4. Scroll down. Commit changes → leave the default message or write your own → choose "Commit directly to main".
  5. Vercel picks it up automatically. ~30 sec to live.

Add a new file

  1. Navigate to the folder where the file belongs.
  2. Top-right → Add file → Create new file.
  3. Type a filename. Paste content. Commit.

Pull requests (optional)

If a change feels risky, commit it to a new branch instead of main (GitHub UI offers this option at commit time). Open a Pull Request. Vercel builds a preview URL on the PR — test it → merge when happy → main rebuilds.

Repo secrets

Settings (gear icon) → Secrets and variables → Actions. Currently set:

VERCEL_DEPLOY_HOOK_URL
URL that the GitHub Action POSTs to on every push, triggering a Vercel build. Don't paste this anywhere public — it's effectively a write key to your Vercel project.

6. Common recipes

Fix a typo on a calculator page

  1. Open the calc's page.tsx on GitHub.
  2. Pencil ✎ to edit. Find the typo. Fix it.
  3. Commit changes → Commit directly to main → green button.
  4. Wait ~30 sec. Refresh the live page.

Add Google Analytics

  1. analytics.google.com → create property → copy the Measurement ID (G-XXXXXXXXXX).
  2. Vercel → Settings → Environment Variables → Add → Key NEXT_PUBLIC_GA_MEASUREMENT_ID → Value G-XXXX → tick Production → Save.
  3. Deployments tab → ⋯ on latest → Redeploy.
  4. Wait ~30 sec. GA Realtime should show traffic when you visit the site.

Submit the site to Google Search Console

  1. search.google.com/search-console → Add property → URL prefix → paste the production URL.
  2. Choose HTML tag → copy the content="…" value.
  3. Vercel → Add env var NEXT_PUBLIC_GOOGLE_SITE_VERIFICATION = the value → Save → Redeploy.
  4. Back in GSC → click Verify.
  5. GSC → Sitemaps → enter /sitemap.xml → Submit.

Promote a planned calculator to live

  1. Open data/calculators.json on GitHub.
  2. Find the slug. Change "status": "planned" to "status": "live".
  3. Commit. ~30 sec later it appears on the homepage, in its category page, and in the sitemap.

Add a new FAQ to an existing calculator

  1. Open the calc's page.tsx on GitHub.
  2. Find the FAQS array. Copy an existing { question, answer } block.
  3. Paste it inside the array, edit question + answer text. Keep the trailing comma after the closing brace.
  4. Commit. The visible FAQ AND the FAQPage JSON-LD update automatically.

Buy and connect a custom domain

  1. Buy at Namecheap / Porkbun / Google Domains.
  2. Vercel → Settings → Domains → Add → type the domain.
  3. Copy the DNS records Vercel shows → paste them at your registrar.
  4. Wait 5–60 min for DNS to propagate. Certificate issues automatically.
  5. Update env var NEXT_PUBLIC_SITE_URL to the new domain. Redeploy.
  6. Update GSC + Bing + GA with the new domain.

Roll back a bad deploy

  1. Vercel → Deployments tab.
  2. Find a previous green Ready row from when the site was working.
  3. Click ⋯ → Promote to Production.
  4. Live in ~2 sec. Fix the bug, push a new commit when ready.

Add a redirect from an old URL

  1. Open next.config.ts on GitHub.
  2. Inside the nextConfig object, add a redirects() async function (see template below).
  3. Commit. After deploy, the old URL 308-redirects to the new one.
async redirects() {
  return [
    { source: "/old-calc", destination: "/new-calc", permanent: true },
  ];
}

7. Troubleshooting

Vercel deploy is BLOCKED (red)

Why: The commit author email isn't recognised by Vercel.

Fix: Make a new commit via the GitHub web UI (it uses your GitHub primary email, which matches Vercel).

Vercel build failed with a syntax error

Why: A missing comma, quote, or brace in a .tsx file.

Fix: Vercel → Deployments → the failed row → Build Logs. The error names a file and line number. Open that file on GitHub, fix the punctuation, commit.

Sitemap or JSON-LD shows the wrong domain

Why: NEXT_PUBLIC_SITE_URL env var isn't set, or wasn't applied to Production.

Fix: Vercel → Settings → Environment Variables → confirm the value (no trailing slash) → tick Production → Redeploy.

Page returns 404 NOT_FOUND on Vercel

Why: No deployment exists yet, or the file path doesn't match the URL.

Fix: Confirm the slug in data/calculators.json matches the folder name under src/app/.

Site shows "Authentication Required" page

Why: Vercel Deployment Protection is on.

Fix: Vercel → Settings → Deployment Protection → set to None for production. The protection blocks Googlebot — turn it off for any indexable site.

GA / GTM isn't loading

Why: Env var name typo, or wasn't applied to Production scope, or you forgot to redeploy.

Fix: Confirm the env var name starts with NEXT_PUBLIC_ exactly as in this guide. Tick Production. Redeploy. Hard-refresh the live page (Cmd+Shift+R).

8. Glossary

commit
A snapshot of every file in the repo at a moment. Each commit gets a unique short ID (7 hex chars).
branch
A named line of commits. The live site builds from main.
push
Upload your local commits to GitHub.
webhook
An automatic HTTP POST GitHub sends to Vercel when you push.
env var
A name=value pair Vercel injects into the build. Used for header codes, API keys, the production URL.
canonical URL
The single official URL for a page. Tells search engines which version to index when duplicates exist.
JSON-LD
Structured data Google reads to build rich SERP cards. Emitted as a <script type="application/ld+json"> on every page.
ISR
Incremental Static Regeneration — Vercel can rebuild a page periodically without a full deploy. We don't use it yet; all pages are fully static.
MD3
Material Design 3 — Google's design system. Both the live site and this guide use it.
hydration
React attaching event listeners to the pre-rendered HTML on first page load.

9. Files NOT to touch (without Claude)

src/app/layout.tsx
Wires global metadata, analytics scripts, and the search-console verification slot. Change behaviour via env vars instead.
src/components/md3/
Material Design 3 primitive components. Editing affects every page.
src/lib/site.ts / src/lib/schema.ts
Single source of truth for site config + JSON-LD generators. Touch only with care.
next.config.ts
Next.js config — Turbopack, redirects, headers. Small edits OK once you know what each block does; ask Claude first.
package.json / package-lock.json
Dependencies. Wrong edits break the build entirely.
.github/workflows/
CI workflows. Currently one file there pings Vercel on push. Don't edit unless you're replacing the deploy mechanism.
node_modules/ and .next/
Auto-generated. They're git-ignored anyway, but if you ever see them in a commit diff, something's wrong.
